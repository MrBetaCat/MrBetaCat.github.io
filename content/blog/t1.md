# The Chassis Problem: What I Learned Building the Minimal Agent Skeleton

The first mistake is almost universal. You get access to an LLM API, you write a system prompt, you call the model, you get back a multi-step plan that sounds exactly like what an agent would produce — and you ship it. You have not built an agent. You have built an expensive, single-shot text generator dressed up in agent vocabulary.

The distinction matters because the failure modes are completely different. A text generator fails by producing bad text. An agent fails by taking bad actions in the world, running indefinitely, exhausting your token budget, corrupting downstream state, or silently returning a non-answer that looks like an answer. None of these failures are detectable if you never built the loop in the first place. You cannot break what you never had.

---

## The Engineering Decision

The core decision in building a minimal agent is not which model to use or which framework to reach for. It is: **what is harness code and what is model code, and where exactly is the boundary?**

This sounds obvious until you try to draw it. In practice, engineers consistently push too much responsibility into the model — stopping conditions, error recovery, loop termination, state tracking. The model is a capable reasoner but it has no concept of "enough attempts," no ability to enforce its own iteration limits, and no mechanism to recover from an exception it never sees. These are chassis responsibilities.

The car metaphor earns its keep here. An engine is powerful and inert. It cannot steer itself, cannot apply its own brakes, cannot sense that a tire just blew. The chassis — transmission, steering, brakes, sensors — is what makes the engine useful. Strip the chassis and the engine just spins. In agent terms: the harness is not optional scaffolding you add when things get complex. It is the minimum viable structure without which the model cannot act in the world at all.

## What I Built

I implemented the loop twice: first against the Anthropic SDK, then migrated to Ollama running Gemma 3 27B locally. The migration was instructive precisely because it forced me to confront the schema differences explicitly rather than accepting a framework's abstraction.

The MVS is about 60 lines. Five named components, each with a single responsibility: `perceive` (seed the messages list), `build_context` (reassemble state each turn), `reason` (the only line that touches the model), `parse_action` (read `msg.tool_calls`, not model prose), and `execute` (run the tool, return a string). The loop guard is a `for` loop with a `max_iterations` constant — not a prompt instruction, not a model-side heuristic. Harness code.

The non-obvious decision: `execute_tool()` is declared to return `str` with no exceptions. A `try/except` wraps the entire body. This is not defensive programming for its own sake — it is a type contract. The loop expects observations, not crashes. Conflating tool failure (a normal event) with program failure (an abnormal event) is the mistake; the type signature enforces the distinction.

---

## What I Broke (And Why It Matters)

I induced both failure modes deliberately.

First, I removed `max_iterations`. The model called `search_documents` on every iteration because the mock knowledge base returned empty results — the model had no answer and no mechanism to stop seeking one. The loop ran until I killed the process. In production this is not an annoyance; it is a cost incident and potentially a hung user-facing request with no timeout. The fix is structural, not prompt-based. Writing "stop after five attempts" in a system prompt is a suggestion. `for iteration in range(1, max_iterations + 1)` is a law.

Second, I replaced the `return error_string` path in `execute_tool()` with `raise KeyError`. The loop crashed on the first missing document ID with a Python traceback — no recovery, no final answer, no diagnostic output. The model never saw the error; the harness absorbed it as an exception and died. The correct design treats every tool result, including errors, as an observation the model can reason about. The model recovered correctly once I passed it `"Error: doc_id 'doc_999' not found. Valid IDs: ['doc_001', 'doc_002', 'doc_003']"` — it simply tried a valid ID on the next turn.

---

## Connection to Anthropic's Design Principles

Both failure modes map directly to the minimal footprint principle in Anthropic's model spec. The spec states that agents should avoid acquiring resources, influence, or capabilities beyond what is needed for the current task — and that harnesses must enforce this independently of model judgment. A loop without a guard violates minimal footprint structurally: the agent consumes unbounded compute and tokens, not because the model chose to, but because the harness gave it no constraint. The spec is explicit that safety properties must live in the harness, not be delegated to model self-regulation. Building the loop guard as harness code is not a stylistic choice — it is the direct implementation of that principle.
